// 01

What problem are we solving?

When you write math, you use infix notation — the operator sits between the two values it operates on. You've been reading this since grade school and your brain handles it naturally.

Computers, however, need to know the order of operations explicitly. They cannot intuitively "see" that multiplication happens before addition. They need a format with no ambiguity at all — and that format is called postfix notation (or Reverse Polish Notation).

Infix — human-readable
3 + 4 * 2

Requires knowledge of precedence rules. You must know that * runs before +, otherwise you'd get the wrong answer.

Postfix — machine-friendly
3 4 2 * +

No ambiguity. Scan strictly left-to-right: push 3, push 4, push 2, multiply top two (4×2=8), add top two (3+8=11). Done.

Why does this matter in real life?

Compilers, calculators, spreadsheet engines, and expression evaluators all convert infix to postfix (or a similar tree structure) internally before computing. Understanding this algorithm is understanding how machines parse and evaluate expressions.

// 02

Why a stack?

A stack is a last-in, first-out (LIFO) container. The last thing you push onto it is the first thing you get back. Think of a stack of plates — you always take from the top.

The algorithm needs to hold operators temporarily while it decides when to release them. The rule is: an incoming operator should trigger the release of any previously stored operators that have higher or equal precedence. A stack enforces this perfectly — you can only ever look at and remove the most recent thing you stored, which is always the one with the highest pending precedence.

Why not a queue?

Queue is first-in, first-out

A queue would release operators in the order they arrived, ignoring precedence entirely. That gives you wrong output.

Why not an array index?

Random access isn't needed

You never need to reach into the middle. You only ever look at and remove the top element. A stack with O(1) push/pop is the perfect fit.

Why not recursion?

Stack does the same thing

Recursive solutions build an implicit call stack anyway. An explicit stack gives you control and avoids stack overflow on long expressions.

Precedence and associativity

Before looking at any code, you need to understand the two properties that govern every decision the algorithm makes.

+ −
addition / subtraction
weight: 1 (lowest)
* /
multiply / divide
weight: 2 (medium)
$
exponentiation
weight: 3 (highest)
Associativity — which direction do equal operators bind?
L
Left-associative (most operators)
a - b - c means (a - b) - c. Equal-precedence operators pop each other from left to right. When a new operator arrives with equal precedence, the one on the stack pops first.
R
Right-associative (exponentiation only)
a $ b $ c means a $ (b $ c). When a new $ arrives and the stack top is also $, do not pop — let the new one go on top. This reverses the usual rule for equal precedence.
Bug alert — the switch fall-through problem

The GetOperatorWeight function in this code has a classic C++ bug. There are no break statements between cases, so execution falls through:

// What the programmer intended:
case '+': case '-': weight = 1;  // ← missing break!
case '*': case '/': weight = 2;  // ← missing break!
case '$':           weight = 3;

// What actually happens when you call GetOperatorWeight('+'):
// enters case '+', sets weight = 1
// falls through to next case, sets weight = 2
// falls through to '$' case, sets weight = 3
// returns 3 — WRONG!

// Fix: add break after each group
case '+': case '-': weight = 1; break;
case '*': case '/': weight = 2; break;
case '$':           weight = 3; break;
// 04

The four rules (Shunting-yard algorithm)

The algorithm scans the input from left to right. For each character it encounters, exactly one of these rules applies:

1
Operand (letter or digit)
Write it directly to the output. Operands are never held — they flow straight through. This is why operands appear in the output in their original left-to-right order.
2
Operator (+, −, *, /, $)
While the stack is not empty, the top is not (, and the top operator has higher (or equal + left-assoc) precedence over the incoming one — pop it to output. Then push the new operator. This is where precedence is enforced.
3
Left parenthesis (
Push immediately onto the stack. It acts as a barrier — nothing inside the parentheses can pop past it. It has the lowest effective precedence when on the stack.
4
Right parenthesis )
Pop and output everything until the matching ( is found. Then discard both parentheses — they were just scope markers and don't appear in postfix output at all.
✓
End of input — flush the stack
Pop and output every remaining operator on the stack. These are lower-precedence operators that were waiting for something to trigger their release — now there's nothing left to wait for.

Every function in the code — explained

Click any function to expand its explanation.

IsOperand(char C)
Character classifier
▾
Returns true if the character is a digit 0–9 or a letter a–z or A–Z. It uses three ASCII range comparisons. This is the simplest guard in the algorithm — if it's a letter or digit, it's a value, not an instruction.

Limitation: This only handles single-character operands. Multi-digit numbers like 42 would be treated as two separate operands 4 and 2. Real parsers handle this with a tokenizer that groups consecutive digits first.
IsOperator(char C)
Character classifier
▾
Returns true only for the five expected operators: +, -, *, /, and $. This is an explicit whitelist — nothing slips through by accident. The $ symbol is used here as a stand-in for exponentiation (often written ^ in math, but ^ is the XOR operator in C++ so a different symbol was chosen).
GetOperatorWeight(char op)
Precedence lookup — contains a bug
▾
Meant to return 1 for +/-, 2 for */, and 3 for $. However, the missing break statements cause all operators to fall through to weight 3 (only $ actually returns correctly because it's the last case before the closing brace sets the final value). In practice, the algorithm still runs but precedence between +/- and */ is not properly enforced.

This is one of the most common C++ beginner mistakes. The fix is simple: add break; after each case group.
IsRightAssociative(char op)
Associativity check
▾
Returns true only for $ (exponentiation). All other operators are left-associative. This function is consulted by HasHigherPrecedence to handle the special case where two operators have equal weight — normally you pop, but for right-associative operators you do not.

Mathematically: 2 ^ 3 ^ 2 means 2 ^ (3^2) = 2^9 = 512, not (2^3) ^ 2 = 8^2 = 64.
HasHigherPrecedence(char op1, char op2)
Core decision function
▾
This is the brain of the algorithm. op1 is the operator on top of the stack; op2 is the one just arrived from the input. It answers the question: "Should op1 be popped before op2 is pushed?"

The answer is yes (true) if:
— op1 has strictly higher weight than op2, OR
— they have equal weight AND op1 is left-associative.

The answer is no (false) if op1 has lower weight, or if they are equal weight and op1 is right-associative (like $).
InfixToPostfix(string expression)
Main conversion function
▾
The top-level function. It owns the stack and the output string. It scans the expression character by character, skips spaces and commas, and applies the four rules using all the helper functions above. At the end, it flushes any remaining operators off the stack into the output.

This is a direct, clean implementation of Dijkstra's Shunting-yard algorithm. The time complexity is O(n) — every character is processed at most twice (once when read, once when popped from the stack). Space complexity is O(n) for the stack in the worst case.

Why did − pop both * and + ?

When − arrives, the stack top is *. Since − has weight 1 and * has weight 2, HasHigherPrecedence(*,−) returns true → pop *. Now the new top is +. Since − and + have equal weight 1 and both are left-associative, HasHigherPrecedence(+,−) returns true → pop +. Now the stack is empty, so − is pushed.

// 08

Bugs, limitations, and design choices

Bug

Missing break in switch

GetOperatorWeight falls through all cases, giving every operator weight 3. The fix: add break; after each case block.

Limitation

Single-character operands only

Multi-digit numbers like 12 are treated as two separate tokens. A real tokenizer would group consecutive digits first.

Limitation

No parenthesis mismatch detection

An expression like A+(B silently produces wrong output. A robust implementation checks for unmatched parentheses and throws an error.

Design choice

Spaces and commas as delimiters

Spaces and commas are explicitly skipped. This is intentional — it allows more human-readable input without breaking the algorithm.

Design choice

$ for exponentiation

^ is the bitwise XOR operator in C++. Using $ avoids operator collision while still demonstrating right-associativity.

Complexity

O(n) time, O(n) space

Each character is processed at most twice — once on read, once on pop. Stack size grows at most to the number of operators. This is optimal.
